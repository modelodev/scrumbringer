# Auditoria general de calidad y plan de refactorizacion

Fecha: 2026-06-05
Alcance: monorepo completo (`apps/client`, `apps/server`, `shared`, `packages/birl`, `packages/lustre_http`, docs tecnicas y tests).

## Resumen ejecutivo

El proyecto ya ha avanzado en modularizacion y la rama de refactor ha cerrado varios bloqueos del informe inicial: el cliente vuelve a compilar, los tests ya no usan `gleeunit/should`, no quedan `let assert`/`panic`/`todo`/`unwrap` en codigo de produccion inspeccionado, y los contratos compartidos principales rechazan estados invalidos en vez de normalizarlos. `CardColor`, `InviteLinkState`, `RuleTarget`, roles, estados de tareas/cards/milestones y varios contratos de URL/storage/API ya usan ADTs o parsers estrictos en owners canonicos, eliminando parsers/listas duplicadas en UI, payloads y persistencia.

La complejidad accidental que queda es principalmente estructural: los coordinadores cliente siguen siendo grandes (`client_view`, `client_update`, `features/pool/update`, `features/admin/update`, `features/milestones/update`, `features/tasks/update`) y todavia contienen adaptacion al `Model` global. HTTP ya tiene payloads/presenters para los handlers principales; el foco restante es conservar ese patron, no reintroducir JSON/Dynamic en coordinadores. Persistencia ya valida muchas filas mediante mappers y helpers pequenos, pero debe mantenerse como disciplina para cualquier dominio nuevo o no tocado.

La simplificacion mas valiosa no es seguir partiendo archivos por tamano, sino reforzar ownership y tipos:

1. Seguir reduciendo coordinadores cliente mediante submodelos reales y owners locales de update/view.
2. Mantener Parse -> Process -> Present en HTTP y tratar cualquier regreso de `Dynamic`/`json` al handler como regresion.
3. Encapsular filas generadas SQL en mappers que devuelvan `Result` antes de entrar en logica de dominio.
4. Consolidar contratos compartidos restantes con parsers estrictos y ADTs canonicos.
5. Reducir duplicacion de CRUD dialogs con helpers concretos, evitando abstracciones genericas prematuras.
6. Mantener gates rapidos por paquete y reservar `make test` como gate final con DB local.

## Hallazgos principales

### Resuelto: el cliente vuelve a compilar

Ubicacion original: `apps/client/src/scrumbringer_client/features/tasks/update.gleam`.

El bloqueo inicial por referencias a `detail_editor.*` ya no se reproduce. `gleam check` en `apps/client` pasa en la rama de refactor.

Accion restante: mantener `apps/client gleam check` como gate obligatorio antes de cada migracion estructural de frontend.

### Major: la modularizacion frontend es superficial en varias features

Ubicacion: `apps/client/src/scrumbringer_client/client_state.gleam:240`, `:280`; `features/pool/update.gleam:1297`; `features/admin/update.gleam:923`; imports globales en multiples `features/*`.

Aunque hay `features/<area>/state/msg/update/view`, gran parte de las features importa `client_state.{type Model, type Msg}` y muta el root model. Esto mantiene el acoplamiento central, obliga a conocer todo el estado desde cada feature y hace que los dispatchers sigan creciendo.

Accion: mover cada feature a `update(feature_model, feature_msg, deps) -> #(feature_model, Effect(parent_msg))`. El root debe ensamblar y enrutar, no ser el estado que todas las features modifican directamente.

## Revision modulo a modulo

### Root, build y DB

`Makefile` tiene targets razonables (`build`, `test`, `verify`, `squirrel`), pero `make test` mezcla migracion DB y todos los paquetes. Para refactors grandes conviene usar gates por paquete primero y `make test` como gate final. Migraciones SQL estan atomizadas y no destacan como foco principal de complejidad.

### `shared`

Es el lugar correcto para ADTs de dominio y codecs, pero aun hay parsers tolerantes que esconden errores. Prioridad: hacer estrictos `TaskState`, `CardState`, `MilestoneState`, roles y workflows; anadir round-trips JSON y mappers DB -> domain en tests.

### `apps/client` core

`client_state.gleam`, `client_update.gleam`, `client_view.gleam`, `router.gleam` y `url_state.gleam` siguen siendo hubs. El plan previo queria que fueran ensambladores; todavia contienen reglas de negocio de navegacion, hydration y filtros. La simplificacion debe mover reglas a owners locales y dejar root como coordinador. `router.apply_mobile_rules` se elimino porque ya era una identidad de compatibilidad; el layout mobile se expresa en `ui.is_mobile` y `parse_uri` conserva directamente la ruta `Pool` en mobile y desktop. El reset de drag state ya no esta reescrito en las ramas de navegacion ni en los retornos 401 de admin/pool: `client_update.gleam`, `features/admin/update.gleam` y `features/pool/update.gleam` delegan en `client_state/member.reset_drag_state`, dejando las copias manuales solo en el owner de estado inicial y en el flujo propio de drag/drop. El adapter `features/admin/view_configs.gleam` se elimino porque solo era consumido por `client_view.gleam`; los builders root-aware de Invites, Projects, Assignments y Metrics viven ahora como helpers privados del ensamblador raiz junto al resto de callbacks admin.

`pool`, `tasks`, `admin`, `milestones`, `assignments`, `projects` y `auth` concentran la complejidad. El patron actual "feature module con root Model" limita el beneficio de la separacion. `view_form_error` centraliza el bloque compacto `form-error` usado por `task_type_crud_dialog` sin mezclar sus estilos locales. `view_submit_button` y `view_danger_button` centralizan los botones de submit y delete de esos cuatro CRUD dialogs principales, manteniendo los labels traducidos en cada owner y evitando un builder generico de footer. `view_primary_action_button` y `view_danger_action_button` cubren los botones por mensaje de `task_type_crud_dialog`, incluido el guard extra que impide borrar tipos con tareas. `task_type_crud_dialog.view_optional_fields` elimina la duplicacion local del bloque icono + capability entre create/edit, y `view_name_field` hace lo mismo con el campo obligatorio de nombre, sin convertir ninguno de los dos en una abstraccion global de formulario. `task_template_crud_dialog.view_template_fields` consolida los campos compartidos de nombre, descripcion, tipo, prioridad y hint de variables entre create/edit, manteniendo los callbacks especificos de cada formulario en vez de introducir un builder generico. `workflow_crud_dialog.view_workflow_fields` aplica el mismo patron local para nombre, descripcion y activo, preservando aria labels especificos de create y callbacks propios de create/edit. `card_crud_dialog.view_card_fields` consolida titulo, descripcion y color entre create/edit, dejando el contexto de milestone como campo exclusivo de create y conservando autofocus/aria labels solo donde existian. `rule_crud_dialog.view_rule_fields` consolida nombre, objetivo, tipo de recurso, tipo de tarea condicional, estado destino y activo entre create/edit, manteniendo selectores dinamicos y aria labels locales al componente.

Nota adicional de avance: los comentarios genericos `Justification: nested case improves clarity` y `large function kept intact to preserve cohesive UI logic` se eliminaron de produccion. Las notas `Justification` restantes describen una decision concreta, como scoping de dialogos, constraint names, reintentos por token unico o diferenciacion de estados de metricas.

### `apps/client/api`

La API cliente esta mejor dividida por dominio. Debe depender de `shared/domain/*/codec` siempre que el contrato sea compartido, y reservar decoders locales para payloads estrictamente de transporte UI.

`api/org` ya no acepta strings libres para roles de proyecto al anadir/actualizar proyectos de usuario; recibe `ProjectRole` y serializa a string solo en el cuerpo HTTP. Las fachadas `scrumbringer_client/decoders.gleam` y `api/tasks/decoders.gleam` ya no existen; el cliente debe importar los codecs compartidos directamente.

El cuerpo se lee despues de pasar los gates de acceso, `create_invite` recibe `CreateInvitePayload` tipado y el mapper de errores mantiene `FORBIDDEN`, `Invalid JSON`, `expires_in_hours must be positive` y `Database error`.

### `apps/server/services`

`services/workflows/handlers.gleam` es el centro de negocio de tareas y debe seguir existiendo, pero con helpers mas pequenos y errores de dominio mas especificos. Tambien concentra defaults de ejecucion en helpers privados (`event_task_type_label`, `event_from_state_label`, `template_description_text`, `task_type_filter_value`, `card_id_create_value`, `no_task_milestone_id`) en vez de mezclar `Option.unwrap` o sentinels posicionales con logging, busqueda de reglas y sustitucion de plantillas. `rules_db.gleam` ya no usa asserts de mapeo: separa `parse_stored_target` para datos persistidos corruptos (`Unexpected("Invalid persisted rule target")`) y recibe targets de create/update ya parseados por `http/rules` como `workflow.RuleTarget`; `update_rule_with_row` usa un helper local para defaults de `Option` en vez de repetir seis ramas.

### `apps/server/persistence` y `sql`

No editar `sql.gleam` manualmente. La deuda real es construir una capa anti-corrupcion: SQL row -> validated domain -> service. Las filas generadas no deberian escapar a HTTP ni a logica de negocio.

Nota adicional de avance: `services/task_templates_db.gleam` ya valida `priority` tambien al leer filas persistidas. Los mappers `from_*_row` devuelven `Result(TaskTemplate, ServiceError)`, la lista usa `list.try_map`, y la corrupcion de DB se representa como `Unexpected("Invalid persisted task template priority")` en vez de construir un `TaskTemplate` invalido. El handler HTTP de listado consume ahora `ServiceError` con el mismo mapper que create/update/delete. `db_error_paths_test.gleam` anade un caso que corrompe una fila a `priority = 0` y espera ese `Unexpected`; no se pudo ejecutar la suite server en este entorno porque falta `DATABASE_URL`, pero `gleam check`, `gleam format --check` y `git diff --check` pasan.

Nota adicional de avance: `services/rules_db.gleam` aplica el mismo patron a las plantillas asociadas a reglas. `list_rule_templates` devuelve ahora `Result(List(RuleTemplate), ServiceError)`, convierte errores SQL a `DbError`, usa `list.try_map` y rechaza `priority` persistida fuera de 1..5 como `Unexpected("Invalid persisted rule template priority")`. Los consumidores HTTP (`list_rules_with_templates` y `attach_rule_template_db`) mantienen el contrato publico usando `service_error_response.to_database_response`. `db_error_paths_test.gleam` cubre el caso corrompiendo la prioridad de una plantilla adjunta antes de listar las plantillas de la regla.

Nota adicional de avance: `services/rules_engine.gleam` valida tambien la prioridad de las plantillas que va a ejecutar. `get_rule_templates` usa ahora `execution_template_from_row -> Result(ExecutionTemplate, RuleEngineError)` con `list.try_map`; una prioridad persistida invalida corta la evaluacion como `InvalidRuleTarget` antes de crear tareas hijas. `unit/services/rules_engine_test.gleam` cubre el caso corrompiendo `task_templates.priority` despues de adjuntar la plantilla y esperando `Error(InvalidRuleTarget)` al llamar `evaluate_rules`.

Nota adicional de avance: los errores de persistencia ya preservan el dato corrupto o el contexto de operacion. `services/persisted_field.gleam` centraliza patrones pequenos: `required`/`optional_blank` para parsear strings persistidos con `Unexpected("...: " <> value)`, `returned_row` para operaciones `RETURNING` que no devuelven fila con `Unexpected("<operacion> returned no row")`, `query_row` para funciones que ya devuelven `pog.QueryError`, y decoders escalares de columna 0 para `pog.returning`. Con esto desaparecen los mensajes genericos `Unexpected("empty_result")`, `Unexpected("no_row_returned")`, los decoders locales triviales `int_field_decoder`/`bool_field_decoder`/`string_field_decoder` y varios defaults silenciosos (`0`, `False`, `""`) de queries que por SQL deben devolver una fila. `persistence/auth/queries`, `projects_db`, `cards_db`, `metrics_db`, `rule_metrics_db`, `org_users_db`, `password_resets_db`, `work_sessions_db`, `capabilities_db`, `task_types_db` y los stores de vistas task/card usan ahora ese helper donde corresponde. Los mappers de tareas, roles, plantillas y reglas tambien incluyen ahora el valor invalido o los campos de target persistidos en el mensaje, evitando perder informacion de diagnostico.

Nota adicional de avance: `services/task_types_db.gleam` valida tambien el sentinel SQL de `capability_id`. Las filas `TaskType*Row` ya se ensamblan por `Result`: `0` sigue representando ausencia, IDs positivos se convierten a `Some(id)`, y cualquier entero negativo se rechaza como `Unexpected("Invalid persisted task type capability_id: <id>")`. `list_task_types_for_project` devuelve ahora `ServiceError` para poder propagar corrupcion de mapper igual que create/update, y `workflows/handlers.gleam` mapea ese `ServiceError` de forma explicita en el flujo `ListTaskTypes`.

### `packages/lustre_http`

El paquete es pequeno y util. La simplificacion de `expect_json` ya esta aplicada: la respuesta HTTP se normaliza en `response_to_result`, y el JSON se decodifica en `json_response_to_result` con `result.try` + `json.parse`, sin anidar casos de exito/error en la API publica. La validacion actual es `gleam check`; el test del paquete sigue siendo smoke porque `Expect` es opaco y los helpers son privados.

### `packages/birl`

Es un paquete vendorizado grande, pero la deuda marcada previamente sobre asserts en parsers publicos ya no se reproduce: la busqueda actual en `packages/birl/src` no encuentra `let assert`, `panic`, `todo`, `result.unwrap`, `option.unwrap` ni `lazy_unwrap`. La validacion actual es `gleam check` y `gleam test` (6 tests). Si se toca como libreria publica, mantener parsers fallibles como `Result` y ampliar tests de parseo/calendario en el mismo cambio.

## Plan de refactorizacion

### Fase 0: baseline de refactor

Objetivo: mantener proyecto compilable antes de tocar arquitectura.

- Mantener `gleam check` en client/server/shared/birl/lustre_http como gate rapido.
- Ejecutar `make test` con DB local cuando `dbmate` este disponible.
- Registrar cualquier test roto antes de cada fase estructural.
- No iniciar migraciones frontend grandes si alguno de los checks rapidos falla.

### Fase 1: contratos compartidos estrictos

Estado actual: muy avanzada. El objetivo no es abrir una nueva pasada masiva, sino no reintroducir strings de dominio donde ya existe ADT o parser canonico.

- Mantener `state_from_string`/`from_db` como `Result` en entidades ya migradas.
- Mantener `workflow.Rule` con `RuleTarget` tipado dentro del dominio.
- Mantener `CardColor` como tipo compartido canonico; no reintroducir enums/listas locales de colores en UI o payloads.
- Mantener `domain/card/codec.optional_color_decoder` como decoder compartido; no duplicar parseo opcional de color en componentes cliente.
- Mantener `domain/card/codec.card_decoder` como decoder compartido para eventos/propiedades de card en cliente.
- Mantener los custom elements CRUD sobre codecs canonicos de dominio; decodificar localmente solo campos propios de UI como `_mode`.
- Mantener `InviteLinkState` como tipo compartido canonico; no traducir estados de invitacion por strings en cliente.
- Mantener payloads HTTP de roles tipados con `OrgRole`/`ProjectRole`; no parsear roles por segunda vez en handlers.
- Mantener roles de autor de notas como `OrgRole`/`ProjectRole`; serializar a string solo al presentar JSON/UI o al cruzar SQL.
- Mantener parsers de roles compartidos con errores tipados que conserven el valor invalido; colapsar a `Nil` solo en adaptadores locales que ya tengan ese contrato.
- Mantener parsers de `TaskStatus`/`WorkState` con errores tipados; convertir a mensajes solo en boundaries de diagnostico o HTTP.
- Mantener decoders compartidos para estados derivados de payloads con forma DB; no duplicar `TaskState` parsing en cliente o metricas.
- Introducir tipos compartidos solo para contratos restantes que sigan siendo strings de dominio reales, no para valores de formulario o labels UI.
- Mover codecs restantes a `shared/domain/*/codec` cuando haya mas de un consumidor o el contrato cruce cliente/servidor.
- Anadir tests de parse failure y JSON round-trip en el mismo cambio que toque el contrato.

### Fase 2: feature ownership en cliente

Estado actual: avanzado en subflujos concretos, pero sigue siendo el principal frente estructural. El objetivo no es eliminar toda mencion de `client_state.Model`, sino confinarla a ensambladores/coordinadores y bajar transiciones puras al owner local.

- Mantener el patron ya probado: `FeatureModel`, owner `try_update`, `Effect(parent_msg)`, politica explicita para auth/root/focus/core cuando haga falta.
- Seguir extrayendo de `pool`, `tasks`, `admin`, `milestones` solo flujos con frontera clara y tests locales.
- Eliminar imports directos de `client_state.Model` en feature modules salvo ensambladores.

### Fase 3: simplificar routing, URL y hydration

Estado actual: parsers y defaults principales ya se movieron a owners explicitos. Pendiente solo si aparece nueva logica de ruta/query fuera de `url_state`.

- Mantener reglas de query/admin assignments en `url_state`.
- Reducir nueva logica de route a "route -> patch root shell + feature route msg".
- Mantener `hydration/update.gleam` como ensamblador root-aware y extraer solo decisiones puras que tengan tests claros.
- Anadir tests de round-trip parse -> format -> parse para rutas criticas nuevas.

### Fase 4: CRUD dialogs y UI reusable

Estado actual: helpers concretos de header/footer/error/buttons/form blocks ya estan extraidos. El siguiente paso debe ser incremental y local al componente que duplique estructura.

- No introducir un builder generico global de formularios.
- Extraer `DialogLifecycle` solo si dos o mas dialogs necesitan compartir estado, no por simetria.
- Seguir consolidando bloques repetidos dentro de cada dialog antes de moverlos a `crud_dialog_base`.
- Revisar custom elements solo cuando un caller necesite controlar la vista desde el parent.

### Fase 5: backend HTTP Parse -> Process -> Present

Estado actual: los handlers principales ya siguen el patron en gran parte. El objetivo es convertirlo en regla de mantenimiento.

- Mantener `http/tasks` y los handlers ya migrados como referencia.
- Para cualquier endpoint nuevo o tocado, crear parser/validator/presenter local.
- Devolver errores HTTP desde un mapper unico por dominio cuando el mapeo se repita.
- Tests unitarios de parseo y presentacion, mas integration tests existentes.

### Fase 6: persistencia validada

Estado actual: muchos servicios ya usan mappers validados; mantener el patron como contrato de persistencia.

- Crear mappers `row_to_domain(...) -> Result(domain, ServiceError)` para cualquier dominio pendiente o query nueva que devuelva datos persistidos no triviales.
- Usar helpers de campo persistido pequenos para parsers canonicos repetidos (`required`, `optional_blank`) sin ocultar los mappers por dominio.
- Mantener eliminado `let assert` en produccion; no reintroducir asserts sobre datos de DB/API.
- Revisar cardinalidad de queries `returning` con errores tipados.
- Mantener `sql.gleam` generado y regenerable.

### Fase 7: tests y gates de calidad

Objetivo: refactor incremental sin regresiones.

- Mantener la suite sin `gleeunit/should`.
- Anadir tests de contrato compartido antes de cambiar codecs.
- Tests de feature update para cada migracion de submodel.
- Gate por fase: `gleam check`, `gleam format --check`, tests del paquete tocado, `make test` al cierre.

## Orden recomendado

1. P1: cerrar validacion final en entorno con DB (`DATABASE_URL` y `make test`/gate equivalente con `dbmate`).
2. P1: mantener produccion sin `let assert`/`panic`/`todo`/`unwrap` y sin `gleeunit/should` en tests.
3. P2: seguir reduciendo `client_view`, `client_update`, `pool/update`, `admin/update`, `milestones/update` y `tasks/update` por flujos con owner local y tests.
4. P2: aplicar mappers `row_to_domain -> Result` a cualquier persistencia restante o nueva.
5. P2: mantener Parse -> Process -> Present en HTTP; revisar solo endpoints que vuelvan a mezclar JSON/Dynamic en handlers.
6. P3: continuar CRUD dialogs con helpers concretos y locales, sin builder generico.
7. P3: mantener docs vivas como estado actual + decisiones vigentes.

## Riesgos

- Refactor frontend sin baseline compilable: alto riesgo. No empezar fases estructurales si `gleam check` deja de pasar.
- Cambiar parsers compartidos a `Result`: puede revelar errores ocultos en DB/API. Mitigar con tests de contrato y migracion por entidad.
- Abstraer CRUD dialogs demasiado pronto: riesgo de genericidad opaca. Mitigar extrayendo helpers concretos primero.
- `make test` depende de DB: mantener tambien gates rapidos por paquete para no bloquear iteracion.
