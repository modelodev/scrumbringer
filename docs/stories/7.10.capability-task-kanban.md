# Story 7.10: Vista kanban de tareas por capacidad

## Status

Ready

## Story

**As a** miembro del proyecto,
**I want** una vista tipo kanban donde las tareas activas se agrupen por capacidad,
**so that** pueda visualizar rapidamente la carga y el estado operativo de cada capacidad sin perder el codigo de color de la ficha padre.

## Problem Statement

Hoy el producto permite ver el Pool, el kanban de fichas, Personas y Hitos, pero no existe una vista centrada en **tareas activas por capacidad**. Eso obliga a reconstruir mentalmente el estado de cada capacidad a partir de varias pantallas y dificulta detectar rapidamente donde hay trabajo pendiente, reclamado o ya en curso.

## Scope

**Incluye:**

1. Un nuevo modo de vista de trabajo dentro del flujo actual `router.Member(member_section.Pool, state)` usando `view=capabilities` (sin ruta paralela nueva).
2. Una vista cliente con filas por capacidad que solo muestra tareas activas del proyecto actual.
3. Reutilizacion del componente/patron visual de tarea existente para mantener icono de tipo, titulo, acciones disponibles y borde/color de ficha.
4. Estados vacios, i18n, accesibilidad y pruebas de render/routing del nuevo modo.

**No incluye:**

1. Cambios de dominio o persistencia en servidor.
2. Nuevas reglas de negocio de claim/release/complete.
3. Un rediseño global del sistema de vistas de trabajo.

## Acceptance Criteria

1. Existe una nueva vista **Capacidades** dentro de TRABAJO y la navegacion reutiliza el patron existente de `ViewMode` y `url_state` (`view=capabilities`) sin crear `/app/capabilities`.
2. La vista muestra solo tareas activas del proyecto actual, entendiendo activas como tareas no completadas (`available`, `claimed` y las que esten `ongoing` dentro del estado reclamado).
3. Cada fila representa una capacidad que tenga al menos una tarea activa. Si existen tareas activas sin capacidad asociada, se renderizan en una fila explicita **Sin capacidad** para no ocultarlas.
4. Cada tarea se renderiza con el patron visual ya usado en Lista/Kanban/People (`task_item` + `task_color.card_border_class` o equivalente), manteniendo el color de la ficha padre y evitando introducir una tarjeta alternativa ad hoc.
5. Las acciones existentes de la tarea siguen siendo consistentes con el contexto actual: abrir detalle al pulsar la tarea y permitir `Claim` en tareas disponibles sin inventar acciones nuevas.
6. La toolbar del modo `capabilities` evita contradicciones con el agrupado: mantiene busqueda y filtro de tipo, pero no renderiza el filtro por capacidad.
7. La vista incluye estados `loading`, `empty`, `no-results` y `error`, con copy i18n ES/EN y semantica accesible de lista/secciones.
8. El orden visual de filas es estable y determinista por nombre de capacidad; la fila `Sin capacidad` queda al final si aplica.
9. La implementacion queda cubierta con pruebas de routing/view mode, derivacion de filas por capacidad, render de tareas activas y preservacion del color de ficha.

## UX / Visual Target

```text
CAPACIDADES

Backend
┌──────────────────────────────────────────────────────────────┐
│ [🔧] Ajustar auth middleware                                │
│ [🧪] Corregir test de permisos                              │
└──────────────────────────────────────────────────────────────┘

Frontend
┌──────────────────────────────────────────────────────────────┐
│ [🎨] Refinar board de hitos                                 │
│ [🐛] Fix hover en pool                                       │
└──────────────────────────────────────────────────────────────┘

Sin capacidad
┌──────────────────────────────────────────────────────────────┐
│ [📄] Actualizar copy de onboarding                           │
└──────────────────────────────────────────────────────────────┘
```

Reglas visuales:

1. La fila nombra la capacidad; las tareas viven dentro de esa fila, no en columnas por estado.
2. El estado se comunica por el propio contexto de la tarea y sus affordances existentes, no por un nuevo sistema de colores.
3. El borde/color de ficha padre se mantiene igual que en Pool, Lista, Kanban y People.

### Encaje con la solucion actual

1. **Navegacion:** el repo ya usa `ViewMode` para `Pool`, `Cards`, `People` y `Milestones`; esta historia debe seguir el mismo contrato, no abrir una seccion paralela.
2. **Fuente de verdad:** la vista puede derivarse completamente desde `member.pool.member_tasks`, `member_task_types` y el catalogo de capacidades ya cargado; no requiere cambios backend.
3. **Visual de tarea:** `grouped_list.gleam`, `kanban_board.gleam`, `people/view.gleam` y `pool/view.gleam` ya reutilizan `task_item`, `task_type_icon`, `task_color` y `card_queries`; este modo debe seguir esa misma linea.
4. **Brief fit:** la vista mejora visibilidad del flujo pull sin introducir asignacion directa ni un estado nuevo de dominio.

### Test Design

1. **Routing / URL**
   - parse/format/roundtrip de `view=capabilities`
   - activacion correcta del item de navegacion en sidebar
2. **View derivation**
   - solo tareas no completadas
   - agrupado por `task_type.capability_id`
   - `Sin capacidad` cuando aplique
   - orden determinista de filas
3. **Rendering**
   - uso del borde/color de ficha en tareas con card
   - `Claim` visible en tareas available
   - detalle abrible desde la fila
4. **Toolbar / states**
   - sin filtro por capacidad en este modo
   - estados `loading`, `empty`, `no-results`, `error`

## Risks / Open Questions

1. Hay que decidir el nombre final del modo (`Capabilities` / `CapabilityBoard`) manteniendo coherencia con labels de UI y naming de ficheros.
2. La vista depende de que el catalogo de capacidades este disponible en contexto member; si hoy solo vive en ciertos flujos de admin, habra que asegurar esa hidratacion sin introducir coupling accidental.
3. Si el numero de tareas activas por capacidad crece, podria requerirse scroll horizontal/vertical afinado, pero eso no bloquea este slice.
